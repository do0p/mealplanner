# CLAUDE.md

## What this is

A personal meal-planning webapp. Upload recipe PDFs (or other formats in future), browse the recipe library, plan a week of meals, and generate a scaled shopping list. Recipe text extraction uses a local Ollama LLM.

## Commands

Use `make` from the repo root (`make help` for full list).

### First-time setup

```bash
make install-venv   # create backend venv + bootstrap pip (Python 3.14 needs this)
make install        # install backend pip deps + frontend npm deps
```

### Daily dev

```bash
make dev            # backend on :8000 + frontend on :4200 (concurrent)
make dev-backend    # FastAPI only
make dev-frontend   # Angular only
```

Backend config lives in `backend/.env` (copy from `backend/.env.example`).
On WSL, set `OLLAMA_BASE_URL=http://172.17.176.1:11434`.

### Tests

```bash
make test           # all
make test-backend   # pytest only
make test-frontend  # Vitest only
```

### Build & deploy

```bash
make build          # build-frontend + build-backend (Docker image)
make deploy         # multi-platform build + push to cloud7:5050
make release        # git tag + deploy (uses VERSION file)
```

`build-frontend` also regenerates `frontend/src/app/version.ts` from VERSION.

## Architecture

**Backend** (FastAPI, Python 3.12 in Docker / 3.14 locally):

```
backend/app/
  main.py          # app, CORS, /health, /version, static SPA mount
  config.py        # pydantic-settings: OLLAMA_BASE_URL, OLLAMA_MODEL, DATA_DIR
  db.py            # SQLModel engine, init_db(), get_session()
  models.py        # SQLModel tables + pydantic API schemas
  dependencies.py  # DI: text registry, OllamaRecipeExtractor, ImportService
  ports/           # TextExtractor, RecipeExtractor (ABCs)
  adapters/        # PdfTextExtractor, ExtractorRegistry, FormatDetection,
                   # OllamaRecipeExtractor, FakeRecipeExtractor
  services/
    units.py           # metric conversion, per-person normalization, rounding
    shopping_list.py   # pure aggregation (no LLM)
    recipe_service.py  # recipe CRUD
    plan_service.py    # plan CRUD + shopping list builder
    import_service.py  # upload, queue, background processing (auto-accept)
  routers/         # recipes.py, plans.py, imports.py
```

**LLM usage — import pipeline only:**
The LLM is contacted exclusively at import time (segmentation → extraction). All serving-time features — recipe browsing, scaling, planner, shopping list — are deterministic DB+pure-function operations. The app starts and runs fully without the LLM; only new imports are blocked when it is unreachable. Extracted recipes are accepted automatically (no manual review step).

**Ingredient normalization:**
Ingredients are stored **per 1 person** (`book_quantity ÷ book_servings`). Scaling to N people = `quantity_per_person × N`. Units are normalized to metric (g, ml, pcs) by `services/units.py` as a safety net over the LLM's own conversion.

**Upload decoupling:**
Uploading a file is LLM-free — it stores the file and creates a `pending` ImportJob. Processing is triggered via POST /imports/process and runs in a FastAPI BackgroundTask. The frontend import page polls the LLM status every 5 s and shows a "Process pending" button when the LLM is reachable.

**Format detection:**
Format is detected from file content (magic bytes) at upload time, not from the client content-type. Only `pdf` is supported now. Adding a new format = write one `TextExtractor` subclass + register it in `dependencies.py`.

**Frontend** (Angular 21, standalone components, signals):

```
frontend/src/app/
  app.ts / app.html / app.scss   # root component + nav bar
  app.routes.ts                  # lazy-loaded routes
  app.config.ts                  # providers: router, HttpClient
  models.ts                      # TypeScript interfaces
  api.service.ts                 # all HTTP calls
  version.ts                     # generated by make build-frontend (gitignored)
  pages/
    recipes/           # recipe library grid with search
    recipe-detail/     # ingredient scaler + verbatim instructions
    import-page/       # upload, job queue, LLM status
    planner/           # 7×3 weekly grid, recipe picker overlay
    shopping-list/     # grouped list with checkboxes + print
```

**Data flow for scaling:**
`quantity_per_person (stored) × people (user input) → scaled quantity → metric rounding`

**Shopping list aggregation:**
`build_shopping_list()` in `services/shopping_list.py` accepts a list of `RecipeSelection` dataclasses (title, people, per-person ingredients), scales and converts to metric, aggregates by (normalized name, unit), and groups by category.

## Key API endpoints

All API routes are under the `/api` prefix to avoid conflicts with Angular's client-side routes (which share paths like `/recipes/:id`).

| Method | Path | Description |
|--------|------|-------------|
| GET | /api/recipes/ | list (default: status=accepted) |
| GET | /api/recipes/{id} | full recipe with ingredients + steps |
| PUT | /api/recipes/{id} | update title / ingredients / steps |
| DELETE | /api/recipes/{id} | delete |
| GET | /api/recipes/{id}/source | serve original uploaded file |
| GET | /api/plans/ | list plans |
| POST | /api/plans/ | create plan |
| PUT | /api/plans/{id} | update name + replace entries |
| GET | /api/plans/{id}/shopping-list | computed shopping list |
| GET | /api/imports/llm-status | live Ollama health probe |
| GET | /api/imports/ | list import jobs |
| POST | /api/imports/uploads | upload file (LLM-free) |
| POST | /api/imports/process | trigger background extraction |
| GET | /api/imports/{id} | job detail + extracted recipes |
| POST | /api/imports/{id}/retry | retry a failed job |
| POST | /api/imports/{id}/abort | abort a processing job |
| DELETE | /api/imports/{id} | delete job + recipes + uploaded file |
| GET | /api/health | health check |
| GET | /api/version | app version + timezone + default_servings |

## Testing

Backend tests use an in-memory SQLite DB and `FakeRecipeExtractor` — no Ollama needed.
Frontend tests use Vitest.
